# v1.1 — Hybrid Photo System Spec

**Status:** Draft — start implementing next session
**Owner:** Colby
**Depends on:** Creatomate integration (already complete, template day1_just_listed rendering successfully)

## Problem statement

Two issues observed during the first Creatomate test render on 2026-04-10:

1. **Orientation mismatch.** Listing photos are almost always shot horizontal (landscape), but Stories and Reels render vertical (1080x1920). Creatomate center-crops by default, chopping off the left and right sides of every photo. The realtor beta user called this out as the #1 visual problem with the product.

2. **Visual variety.** The first 5 photos by sort_order on a real listing turned out to be 5 consecutive shots from the same camera session of similar subjects (e.g. 3 front exterior angles in a row). Pipeline logic is correct — photos are distinct files — but perceptually the reel looks like it's repeating the same shot.

## Solution architecture — hybrid A+C

### Phase 1 — Orientation (next session)

1. **Database:** add `orientation` column to `listing_photos` table. Values: `"horizontal"` | `"vertical"` | `"square"`. Migration should backfill existing rows as `"horizontal"` default.

2. **Upload flow:** auto-detect orientation on upload using the existing `sharp` dependency. Compute `width / height` ratio: >1.1 = horizontal, <0.9 = vertical, else square. Write to the new column.

3. **Upload UI:** add an OPTIONAL second photo upload section titled "Vertical photos (optional — for Stories & Reels)" with helper text explaining it's optional and that horizontal photos will be used with a blur background fill if none are provided. This section goes BELOW the existing primary photo uploader, not replacing it.

4. **Photo selection logic:** update the pipeline photo picker. For each content piece, check the target format:
   - Horizontal format (1080x1080 posts, 1200x630 FB) → pick from horizontal photos as today
   - Vertical format (1080x1920 Stories & Reels) → prefer vertical photos if any exist; fall back to horizontal if none
   - Square format → either works

5. **Creatomate template rebuild:** rebuild the day1_just_listed template in the Creatomate editor to include a **blur background fill layer**. Pattern: for each Photo-N slot, add a second image element behind the main one, same source URL, stretched to fill the full 1080x1920 frame, with a heavy blur filter (~40px) and a dark overlay (~30% black). The main photo sits on top at its true aspect ratio, letterboxed against the blurred background. This becomes the universal fallback whenever a horizontal photo ends up in a vertical slot.

### Phase 2 — Visual variety (deferred — not next session)

Use Claude Sonnet vision API at upload time to tag each photo with a room/feature label (exterior, kitchen, living_room, primary_bedroom, bathroom, outdoor, pool, detail_shot, etc.). Store in a new `photo_tags` column on `listing_photos`. Photo picker enforces variety: no two consecutive reel slots get the same tag. Cost estimate: ~$0.01 per photo × 20 photos = $0.20 per listing upload. Implement after Phase 1 ships and we see whether the orientation fix alone is enough to satisfy the realtor.

### Stories content mix: video + engaging stills

Captured from user note during Phase 1 Step 3.

Current behavior: all 5 stories in a package are still images (branded overlays on listing photos). User observation from realtor beta: stories feel flat and repetitive when they're all static.

Desired behavior:
- At least half of the 5 stories per package should be short video creations (1-5 second looping clips using Creatomate motion on a listing photo, or a Runway generated short)
- The remaining stories that stay as stills should use captions and overlay text that are deliberately entertaining, funny, or engaging — not just property-stats callouts

This is a Phase 2 or separate-spec item because it affects:
- Which Creatomate templates are needed for story slots (new video-story templates required)
- Pipeline logic for splitting the 5 story slots between video and still types
- Claude Haiku prompts for story text (need a distinct "entertaining/playful" voice register for still stories)
- COGS per package (adding video stories adds Creatomate render cost per story)

Dependencies: Phase 1 photo orientation system should ship first so any video stories that use photos have correct orientation handling. Template variety work from the v1.1 roadmap document should also be scoped in parallel since adding story variety is a related concern.

Action: after Phase 1 ships and realtor re-tests, scope this as its own spec document before implementation.

## Phase 1 Step 6 — Orientation-Homogeneous Picker + Vertical Hero + Two-Grid UI

### Upload rules

- Horizontal photos: minimum 20, maximum 40. Required. Horizontal hero selected from these (existing is_hero boolean on listing_photos).
- Vertical photos: either 0 or minimum 5, maximum 40. Optional. If a realtor uploads any verticals, they must upload at least 5 — this is enforced server-side in submitListingForReview and /api/checkout. The UI shows warnings but does not block.
- Vertical hero photo: optional, selected from vertical photos only, only relevant when verticals >= 5. Stored as new vertical_hero_photo_id column on listings table (uuid, nullable, references listing_photos(id) ON DELETE SET NULL). If not explicitly set but verticals exist, defaults to the first vertical photo by sort_order.

### Display rules (listing form UI)

Photos are displayed in two separate grids:

1. "Photos" grid (horizontal + square photos). Numbered. HERO badge on the horizontal hero. "Set Hero" action on hover.
2. "Vertical Photos" grid, displayed below the main grid. Numbered separately. V-HERO badge on the vertical hero. "Set Vertical Hero" action on hover. Only shown if vertical photos exist.

The existing collapsible "Vertical Photos" upload drop zone remains, but uploads from it populate the vertical grid below, not the main grid.

### Picker rules (orientation-homogeneous)

Each of the 14 content pieces is classified and filled by orientation. Mixing horizontal and vertical photos within a single reel or story is never allowed.

- Posts (5 pieces): horizontal/square only. Never use verticals. Day 1 post uses horizontal hero. Posts 2–5 pick from horizontal bucket, fall-through to square.
- Reels (5 pieces): orientation-homogeneous. Each reel is entirely vertical or entirely horizontal.
  - Day 2 reel: if verticals >= 5 unused, all-vertical starting with vertical hero + 4 verticals. Otherwise all-horizontal starting with horizontal hero + 4 horizontals.
  - Reels 2–5: for each reel in calendar order, check if 5 unused verticals remain. If yes, fill all-vertical. If no, fill all-horizontal.
- Stories (4 pieces): orientation-homogeneous. For each story in calendar order, check if 1 unused vertical remains. If yes, vertical. If no, horizontal with blur fill in template.

### Priority rule

When verticals are scarce, reels consume them before stories. Processing order:

1. Day 2 reel (vertical hero + 4 verticals, first dibs)
2. Remaining reels in calendar day order
3. Stories in calendar day order

This ensures a realtor uploading exactly 5 verticals gets one flagship vertical reel rather than 4 ephemeral vertical stories.

### Package-wide reuse rule (updated)

Each photo may be used up to 2 times across the 14-piece package, but never twice within the same piece. Pool photos are picked with a usage count; once a photo has been placed twice anywhere in the package, it's treated as unavailable. Within a single reel, a local piece-level used set ensures no photo appears in two slots of the same reel. Hero fallback to hero.id remains the last resort when a pool has no capacity and is effectively unreachable at minimum upload thresholds.

### Math — horizontal pool coverage

With max-2x reuse, each non-hero horizontal photo provides 2 available fills. Day 1 post uses hero directly (not from pool). Demand: 4 posts + 5 reels×5 photos + 4 stories = 33 pool slots (reels at reelPhotoCount=5; at reelPhotoCount=1 demand is 4+5+4=13).

| Horizontals uploaded | Pool photos (minus hero) | Pool fills (×2) | Demand (5-photo reels) | Margin |
|---|---|---|---|---|
| 20 | 19 | 38 | 33 | +5 |
| 25 | 24 | 48 | 33 | +15 |
| 30 | 29 | 58 | 33 | +25 |
| 35 | 34 | 68 | 33 | +35 |
| 40 | 39 | 78 | 33 | +45 |

Hero fallback is unreachable at minimum 20 horizontal uploads.

### Math — vertical allocation (unchanged)

| Verticals uploaded | Vertical reels | Vertical stories | Horizontal fallback |
|---|---|---|---|
| 0 | 0 | 0 | All reels + stories horizontal with blur fill |
| 5 | 1 (Day 2) | 0 | 4 reels + 4 stories horizontal |
| 9 | 1 (Day 2) | 4 | 4 reels horizontal |
| 10 | 2 | 0 | 3 reels + 4 stories horizontal |
| 14 | 2 | 4 | 3 reels horizontal |
| 15 | 3 | 0 | 2 reels + 4 stories horizontal |
| 24+ | Up to 5 | Up to 4 | None if fully stocked |

At max uploads (80 total), the drag-and-drop sort_order persistence loop performs up to 80 sequential DB updates (~5s wall time). Acceptable for Phase 1 since reordering is a rare user action, but batch-update refactor deferred to Phase 2.
